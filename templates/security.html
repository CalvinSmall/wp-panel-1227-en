Fail2ban Configuration

  • Trigger Threshold (times)

    When a single IP triggers the detection rules this many times within the statistical window, it is automatically banned. Default is 5. Lower values are more sensitive (higher risk of false bans), higher values are more lenient. See the "Security Defense" page for specific detection rules.

  • Time Window (seconds)

    The time range used to accumulate violation counts. Default is 60 seconds. Works together with the trigger threshold: for example, the default configuration means "ban after 5 violations within 60 seconds." A larger window makes it easier to trigger accumulation.

  • Ban Duration (seconds)

    How long the IP is blocked by the firewall after the first violation triggers a ban. Default is 600 seconds (10 minutes). After expiration, fail2ban automatically unbans, but if the IP violates again within 24 hours, it is automatically escalated to a 24-hour ban.

Whitelist Mechanism

  • Official Whitelist

    The panel automatically fetches the latest IP ranges from the following official sources and adds them to the fail2ban whitelist to prevent false bans on legitimate crawlers and CDN nodes:

    • Cloudflare — Global CDN and reverse proxy IP ranges (ipv4 + ipv6)
    • Googlebot — Google search crawler official IP ranges
    • Bingbot — Bing search crawler official IP ranges

    After enabling "Weekly Auto-Update," the system automatically re-fetches the latest IPs every Monday at 4:00 AM.

  • Custom Whitelist

    Manually add IPs or IP ranges that should never be banned, one per line. Supports IPv4/IPv6 addresses and CIDR format (e.g., 10.0.0.0/8). Custom whitelist entries are not overwritten by official fetches.

  • WordPress Security Log Path Whitelist

    Used to exclude search engines, ad platforms, site ownership verification, and other legitimate files. This setting only affects WordPress abnormal access statistics, not generic PHP sites, and does not directly allow or block requests.

Rate Limiting Explained

  • How It Works: Token Bucket Algorithm

    The system maintains a token bucket for each IP with a capacity equal to the "Burst Buffer." Tokens are replenished at a rate of 1 per second (determined by the request frequency). Each request consumes a token; when depleted, a 429 is returned. Human browsing has natural pauses (reading pages), during which the bucket refills automatically; bots send continuous requests, so once tokens are depleted, they are limited to 1 request per second.

  • Request Frequency (requests/minute)

    Controls the long-term sustained rate, i.e., token replenishment speed. Default is 60 requests/minute (= 1/second). Bot scanning typically far exceeds this value, so they deplete the token bucket within seconds, and nearly all requests receive a 429.

  • Burst Buffer (requests)

    Controls the token bucket capacity, i.e., the maximum concurrent requests allowed the instant a page opens. Default is 300, sufficient for Elementor + multi-plugin sites (100+ requests per page) opening 2-3 tabs simultaneously. For more concurrent tabs, set a higher value (e.g., 500-600). Non-Elementor / lightweight sites can reduce to 30-120.

    Increasing this value does not delay bot blocking — bots are throttled by the sustained rate (1/second) regardless of burst buffer size; security is not weakened.

  • ⚠ Warning: IPs receiving multiple 429 responses will be automatically banned by Fail2ban

    429 responses are captured by Fail2ban rules and counted as violations (treated at the same level as wp-login brute force attacks). Accumulating violations beyond the threshold will cause the IP to be blocked by the firewall. Ensure the burst buffer is sufficient for daily use to prevent legitimate visitors from accidentally triggering 429 and being blacklisted.

  • ✓ Increasing burst buffer does not affect bot blocking

    Bots continuously send requests (no human browsing pauses), so they are strictly limited by the "Request Frequency" to 1/second. Once tokens are depleted, nearly all requests return 429. Increasing the burst buffer only gives human visitors more headroom for initial page loads; security is completely unaffected. The real limiting factor is the sustained rate, not the burst cap.

  • How to Test if Configuration Is Appropriate

    Open the site in a browser incognito window and randomly browse 5-10 pages (including homepage, category pages, and article pages). Open Developer Tools (F12) → Network tab and check for any red 429 status codes. All 200/304 responses indicate a well-tuned configuration.

Crawler Rate Limiting

  • Rate Limit Scope

    Crawler rate limiting is disabled by default. When enabled, each site has its own independent crawler rate limit bucket; common Bot UAs hitting the same site share this bucket. Normal browsers do not enter this rate limiting layer.

  • Search Engine Exemption

    Googlebot/Bingbot are only exempt from crawler rate limiting when the source IP matches the official IP ranges. Requests impersonating search engines will enter the site's crawler rate limit bucket.

  • Default Recognized Bots

    Currently matched by default: Googlebot, Bingbot, Facebook/Meta (facebookexternalhit, facebook, meta-externalagent), Twitterbot, LinkedInBot, Slackbot, Discordbot, TelegramBot, SemrushBot, AhrefsBot, MJ12Bot, DotBot, and any request with "bot," "crawler," "spider," "scraper," or "scan" in the User-Agent.

  • Recommended Defaults

    Default is 30 requests/minute with a burst of 20, primarily to suppress multi-IP fake crawlers from continuous scanning and reduce WordPress dynamic 404 pressure. After adjusting, monitor the 429 count in access logs and normal crawl behavior.
