Security Defense Rules Explained

Detection Scope

Auto-ban rules monitor access.log and error.log of all sites; WordPress suspicious access analysis additionally reads wp-security.log for local statistics and evidence collection only.

Trigger Conditions (matching any of the following rules counts as 1 violation)

  • POST /wp-login.php — WordPress login brute force attack
  • POST /xmlrpc.php returning HTTP 403 — still being requested after XML-RPC is disabled (including //xmlrpc.php variants)
  • Returning HTTP 429 — exceeding request rate limits
  • Sensitive file scanning returning 404 — e.g., .env, .git, wp-config.php, .sql, archives, etc.
  • 404 flood detection — any path returning 404 up to 30 times/60 seconds is classified as directory scanning (independent rule, fixed threshold)
  • WordPress abnormal path access — logged to wp-security.log for admin analysis, no auto-ban

Ban Threshold

The first 4 rules default to 5 violations within 60 seconds → automatic ban (threshold, window, and ban duration can be customized on the "Security Settings" page).

The 404 flood rule is fixed at 30 times/60 seconds and is not customizable.

Multi-Level Progressive Banning

  • First violation → Fail2ban bans for 10 minutes, auto-unbans on expiry
  • Second violation within 24 hours → escalated to 24-hour ban, also written to Nftables persistent rules
  • 3 cumulative 24-hour bans → automatically converted to permanent ban, enforced via Nftables persistent rules only

The ban list can distinguish between Fail2ban temporary bans and Nftables persistent bans via the "Source" column.

Ban Mechanism

  • Fail2ban — manages website protection bans (jails: wppanel, wppanel-404) and SSH protection bans (jail: wppanel-sshd), auto-releases on expiry
  • Nftables Persistent — manages permanent bans, scan defense bans (source: panel_scan), and panel authentication bans (source: panel), independent of Fail2ban, auto-unbans on expiry
  • Nginx Blacklist — integrated with Fail2ban; when a Web protection jail bans an IP, it is also written to the Nginx geo blacklist, causing banned IPs to receive a 444 (connection closed), ensuring effective blocking even under Cloudflare proxy

All three channels take effect simultaneously: Nftables blocks direct-to-origin requests, Nginx blocks Cloudflare-forwarded requests, and Fail2ban auto-releases on expiry.

Cloudflare Scenario

  • The panel auto-deploys /etc/nginx/conf.d/wppanel-realip.conf to restore the real visitor IP from Cloudflare's CF-Connecting-IP header
  • Whitelist refresh automatically updates Cloudflare official IP ranges, ensuring only Cloudflare sources are trusted
  • No manual configuration needed; automatically takes effect when Cloudflare proxy is detected

To disable Nginx-layer banning: delete /etc/fail2ban/action.d/wppanel-nginx.conf and remove the wppanel-nginx action from /etc/fail2ban/jail.d/wppanel.conf, then run systemctl reload fail2ban && nginx -s reload.

Scan Defense

  • Scan protection for the panel on port 8443; does not affect website ports (80/443)
  • Non-browser characteristic requests (e.g., curl, scripts, scanners) accessing the panel port without matching the random entry → banned for 30 days; common browser probing paths and requests with BasicAuth headers do not directly trigger scan bans
  • Requests matching the random entry (/{suffix}) pass through normally and are not affected

After triggering, the IP is written to the panel ban list with source labeled "Scan Defense" and level set to temporary 30 days.

Whitelist Exemption

127.0.0.1, Cloudflare / Googlebot / Bingbot official IP ranges, and custom whitelist IPs are never banned. The whitelist is managed on the "Security Settings" page.

The WordPress security log path whitelist only excludes legitimate ownership verification files and similar paths; it does not affect the IP ban whitelist.

Rate Limiting Principle

Uses the token bucket algorithm: one bucket per IP, capacity = burst buffer (default 300), replenished at 1 token per second. Each request consumes a token; when depleted, a 429 is returned. Human browsing has natural pauses during which the bucket refills; bots are throttled to 1/second.

⚠ Multiple 429s will be automatically banned by Fail2ban. Ensure the burst buffer is sufficient for daily use.

✓ Increasing burst buffer does not affect bot blocking and does not weaken security.

Server Verification Commands

SSH into the server and run the following commands to check the current ban status:

# Check website protection bans
fail2ban-client status wppanel
# Check 404 flood bans
fail2ban-client status wppanel-404
# Check SSH brute force bans
fail2ban-client status wppanel-sshd
# Check Nftables persistent bans (scan defense + panel auth bans)
nft list set ip wppanel_persist banned_ips
